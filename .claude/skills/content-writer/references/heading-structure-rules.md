# Heading 구조 규칙

## 필수 규칙
| 규칙 | 이유 |
|------|------|
| 페이지당 H1 1개 | SEO 표준. Primary 키워드 포함 |
| H2 = 주요 섹션 | 각각 Secondary 키워드 또는 핵심 질문 타겟 |
| H3 = 하위 섹션 | 긴 H2 섹션을 분리 |
| H1에 Primary 키워드 | 직접적 랭킹 시그널 |
| H2에 Secondary 키워드 | 토픽 커버리지 시그널 |
| 일부 H2는 질문 형식 | PAA(People Also Ask) 타겟 |
| 계층 건너뛰기 금지 | H1 → H3 (H2 없이) 절대 금지 |

## 예시 구조
```
H1: 백엔드 개발자 이력서 작성법 — 경력별 완벽 가이드 (2025)
  H2: 백엔드 개발자 이력서, 왜 일반 이력서와 다를까?
  H2: 경력별 이력서 작성 전략
    H3: 주니어 (1~3년차) — 프로젝트 중심
    H3: 미드레벨 (3~5년차) — 성과 중심
    H3: 시니어 (5년 이상) — 리더십 + 임팩트
  H2: 필수 섹션별 작성 가이드
    H3: 기술 스택 정리하는 법
    H3: 프로젝트 경험 서술 공식
    H3: 자기소개 한 줄 요약
  H2: 흔한 실수 5가지와 해결법
  H2: 실제 합격 이력서 분석
  H2: 자주 묻는 질문 (FAQ)
    H3: 포트폴리오가 없으면 어떻게 하나요?
    H3: GitHub 링크는 꼭 넣어야 하나요?
    H3: 이력서 몇 페이지가 적당한가요?
```
